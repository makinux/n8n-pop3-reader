# Repository Guidelines

## プロジェクト構造とモジュール構成
- ソースノードは `nodes/` に配置され、統合ごとにグループ化されます（例：`nodes/GithubIssues/` で、共有ユーティリティは `shared/` に、操作フォルダは `resources/issue/` に配置）。新しいノードは、`.node.ts` エントリとオプションの `.node.json` UI メタデータを含む独自のフォルダに自己完結的に配置します。
- 認証情報の定義は `credentials/` に配置され、`package.json` の `n8n.credentials` 配下から参照する必要があります。
- 静的アセット（SVG）は、`icons/` または `nodes/<Name>/` 配下のノードの隣に配置されます。ビルド出力は `dist/` に出力されます（git からは除外されます）。
- 設定はリポジトリのルートにあります: `tsconfig.json`、`eslint.config.mjs`、`.prettierrc.js`、`package.json`。

## ビルド、テスト、開発コマンド
- `npm install` — すべての依存関係をインストールします (`@n8n/node-cli` を含む)。
- `npm run dev` — ホットリロードで n8n を起動し、インタラクティブテスト用にローカルノードをロードします。
- `npm run build` — パッケージ化のために TypeScript を `dist/` にコンパイルします。公開前に使用します。
- `npm run build:watch` — n8n を起動せずに変更時にリビルドします。
- `npm run lint` / `npm run lint:fix` — n8n ESLint プリセットを使用して lint (およびオプションで自動修正) を実行します。
- `npm run release` — `n8n-node release` を使用してタグ付けとパッケージ化を行います。`npm run prepublishOnly` は `npm publish` 時に自動的に実行されます。

## コーディングスタイルと命名規則
- 言語: TypeScript。`.prettierrc.js` の Prettier のデフォルト設定（タブ、文字幅 2、シングルクォート、セミコロン、末尾のカンマ、100 文字の行幅）に従ってフォーマットしてください。プッシュする前に、Prettier または `npm run lint:fix` を実行してください。
- `@n8n/node-cli` ESLint ルールに従ってください。インポートはソートし、API ペイロードの型は明示的にしてください。
- Node のフォルダとクラスは PascalCase を使用します（例: `GithubIssues`）。操作には説明的な動詞（`get`、`getAll`、`create`）を使用します。ファイル名はエクスポートされたクラスに合わせてください。
- ID/props は `*.node.ts` と `*.node.json` 間で統一してください。コメントはロジックが明確でない場合にのみ追加してください。

## テストガイドライン
- 自動テストはまだ実装されていません。変更は `npm run dev` を実行し、n8n でノードをロードして、各操作/認証情報パスを実行することで検証してください。
- ワークフローベースのチェックを推奨します。サンプルリポジトリ/問題を使用して、リスト検索 (`listSearch/*`)、リソース操作、エラー処理 (例: 無効なトークン) をテストしてください。
- lint は必須ゲートとして扱い、警告を抑制するのではなく対処してください。

## コミットとプルリクエストのガイドライン
- Git の履歴は最小限に抑えます。分かりやすさを考慮して、Conventional Commit スタイルのメッセージを使用してください (例: `feat: add label filter to GitHub issues`、`fix: guard empty repository input`、`chore: bump node-cli`)。
- プルリクエストには、簡単な概要、テスト内容（コマンドまたはワークフロー手順）、関連する問題（ある場合）を含めてください。動作や設定が変更された場合は、ドキュメント/README を更新し、UI 関連の調整についてはスクリーンショット/GIF を追加してください。
- PR のスコープを限定してください。可能な限り、PR ごとに 1 つの機能/修正を記述してください。ノードや認証情報を追加する際は、`package.json` 配列と `n8n` 配列が同期していることを確認してください。

## セキュリティと設定に関するヒント
- 実際のトークン、PAT、OAuth クライアントシークレットをコミットしないでください。ローカルの `.env` ファイルでは環境変数を使用し、ドキュメントではサンプルのプレースホルダを使用してください。
- 新しい認証情報を追加する場合は、必要なスコープとデフォルトを明示的に指定し、機密性の高いデフォルトをコードに保存しないでください。
- 実行時エラーを回避するため、`shared/transport.ts` の n8n HTTP ヘルパーを使用して外部リクエストを検証し、`shared/utils.ts` で型指定されたレスポンスを優先してください。
